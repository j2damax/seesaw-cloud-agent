# SeeSaw — Privacy Contract

**This document defines what the cloud endpoint receives, what it never receives, and why.**  
**Validate every implementation decision against this contract before writing code.**

---

## The Core Guarantee

> Raw media (pixels, audio) never leaves the iOS device. The cloud endpoint receives **only anonymous semantic labels** and **PII-scrubbed text**. No face data, no bounding boxes, no audio buffers, no device identifiers.

This is enforced structurally in the iOS app by the six-stage Privacy Pipeline (`PrivacyPipelineService`). The cloud endpoint cannot receive raw data even if it tried — it is architecturally impossible given what the iOS `CloudAgentService` sends.

---

## What the Cloud Endpoint Receives

```python
class ScenePayload(BaseModel):
    objects: list[str]           # YOLO label strings, e.g. ["teddy_bear", "book", "sofa"]
    scene: list[str]             # scene categories, e.g. ["living_room"]
    transcript: str | None       # PII-scrubbed speech, e.g. "I love this bear"
    child_age: int               # e.g. 5
    child_name: str              # first name only, parent-provided at onboarding
    story_history: list[StoryTurn] = []   # previous turns (text only)
    session_id: str | None = None         # UUID (pseudonymous)
```

### What these fields contain

| Field | What it is | What it is NOT |
|-------|-----------|---------------|
| `objects` | YOLO class labels (44-class taxonomy) | Pixel coordinates, image crops, face encodings |
| `scene` | VNClassifyImageRequest category strings | Frames, video, depth maps |
| `transcript` | PIIScrubber-redacted speech text | Audio PCM/AAC data |
| `child_age` | Integer from parent profile | Birthdate, date of birth |
| `child_name` | First name, parent-entered at onboarding | Full name, family name |
| `story_history` | Previous story text turns | Any biometric data |
| `session_id` | UUID generated per session | Device ID, Apple ID, IP address |

---

## What the Cloud Endpoint Must NEVER Receive

The following data types are banned from all endpoints, request bodies, headers, and logs:

| Category | Examples | Reason |
|----------|---------|--------|
| Raw image data | JPEG bytes, pixel buffers, Base64 images | GDPR Art.9; COPPA §312.3 |
| Audio data | PCM samples, AAC frames, voice recordings | Biometric identifier under GDPR |
| Face data | Bounding boxes, embeddings, landmark coordinates | Special category data |
| Raw transcripts | Unredacted speech containing names, addresses | PII |
| Device identifiers | IDFA, IDFV, device UUID | Prohibited for under-13 |
| Precise location | GPS coordinates, postcode | COPPA §312.2 |
| IP address (logged) | Any server-side logging of requester IP | GDPR Art.5(1)(c) |

## PII Scrubbing (iOS-Side)

The `PIIScrubber` actor in the iOS app applies regex patterns before the transcript reaches the cloud:

```swift
// Patterns redacted to [REDACTED]:

// - UK/US phone numbers
// - Email addresses
// - UK postcodes, US ZIP codes
// - Credit card patterns
// - UK National Insurance numbers
```

The `totalPiiTokensRedacted` count is stored per session in `StorySessionRecord` and is visible in the Story Timeline UI. **The cloud endpoint should treat the transcript as already scrubbed — do not apply additional PII logic that could log the raw input.**

## Cloud-Side Data Handling Rules

### What to store in Firestore

```
sessions/{session_id}/
  child_age: int
  objects: [str]
  scene: [str]
  created_at: timestamp
  beats/
    {beat_index}/
      story_text: str     ← generated by this service, safe to store
      question: str
      is_ending: bool
      timestamp: timestamp
```

**Do NOT store:** transcript text, child_name (pseudonymise at session creation), IP addresses.

### Logging policy

- **Log at INFO level:** request shape (field count, object count), response generation latency, model used
- **Log at WARNING level:** API key failures, Gemini guardrail hits, Firestore write failures
- **Never log:** transcript content, child_name, session_id in plain text alongside identifying info

### Data retention

Sessions auto-delete after 30 days via Firestore TTL policy:
```python
# Set on every session document
db.collection("sessions").document(session_id).set({
    "ttl": datetime.utcnow() + timedelta(days=30)
}, merge=True)
```

## Compliance Notes

**GDPR (UK/EU):**
- Lawful basis: Legitimate interest (research) + parental consent at onboarding
- Data minimisation: Only semantic labels transmitted (Art.5(1)(c))
- Right to erasure: Session delete via `DELETE /session/{id}` endpoint
- Data processor agreement: Cloud Run in europe-west1 (EU data residency)

**COPPA (US Children's Privacy):**
- No persistent identifiers for children under 13
- Session IDs are UUIDs — not linked to real identity
- No behavioural advertising, no third-party data sharing
- Parent controls story history via the iOS app's Timeline view

**Research Ethics:**
- All data collection approved by [University Ethics Board — citation in dissertation]
- Participants are children aged 3–8 with parental informed consent
- Data used solely for dissertation research, deleted post-viva

## Auditability

The iOS app provides a `rawDataTransmitted: Bool` flag on every `StorySessionRecord` which is hardcoded to `false`. This is visible in the Story Timeline detail view ("Raw data transmitted: No ✓"). Any future change that would set this to `true` would require a deliberate code change and ethics re-approval.

The `totalPiiTokensRedacted` counter per session provides a quantitative measure of PII events caught by the iOS-side scrubber before any cloud transmission.

---

## The Invariant (Do Not Break)

```
∀ session s: s.rawDataTransmitted == false
```

This invariant is the central privacy guarantee of the SeeSaw architecture and the primary research contribution. It must hold for all three story generation modes (onDevice, gemma4OnDevice, cloud).
